## 一、每天开始

### 第一步：同步最新代码 (5分钟)

```bash
# 1. 切换到 develop 分支
git checkout develop

# 2. 拉取最新代码
git pull origin develop

# 3. 如果有冲突，按 GITHUB.md 的方法解决
```

### 第二步：创建/切换到功能分支

```bash
# 如果是新功能，创建新分支
git checkout -b feature/your-feature-name

# 如果是继续昨天的功能，切换到已有分支
git checkout feature/your-feature-name
```

### 第三步：确认今日计划

```
┌─────────────────────────────────────────────────────────────┐
│  在团队群里说一声：                                          │
│                                                              │
│  "今天我要做: xxx功能"                                       │
│  "大概需要: x小时"                                           │
│                                                              │
│  另一个人也说说自己的计划                                     │
│  避免做重复的工作                                             │
└─────────────────────────────────────────────────────────────┘
```

## 二、开发过程中

### 代码提交频率

```
✅ 好习惯: 每完成一个小功能就提交
├── 完成函数A → git commit
├── 完成函数B → git commit
└── 完成测试 → git commit

❌ 坏习惯: 堆积很多代码才提交
├── 工作了一整天
├── 写了几百行代码
└── 最后一次性提交
```

### 提交前检查

```bash
# 1. 查看修改了什么
git status

# 2. 确认没有提交敏感信息
git diff | grep -i "password\|key\|token"

# 3. 运行测试
npm test

# 4. 代码检查
npm run lint

# 5. 提交
git add .
git commit -m "feat: describe what you did"
git push origin feature/your-feature
```

### 需要沟通的时候

```
立即联系对方的情况:
├── 发现接口有问题
├── 需要修改接口
├── 发现对方的Bug
├── 自己搞不定的问题
└── 需要对方帮忙测试

不需要沟通的情况:
├── 修改自己负责的文件
├── 按照接口写代码
└── 日常的bug修复
```

## 三、每天结束

### 第一步：同步代码

```bash
# 推送今天的代码
git push origin feature/your-feature
```

### 第二步：报告进度

```
在团队群里说:

✅ 今天完成:
├── 实现了 xxx 功能
├── 修复了 xxx Bug
└── 完成了 xxx 测试

⏸️ 明天计划:
├── 完成 xxx 功能
└── 开始 xxx 功能

❌ 遇到的问题:
├── xxx 问题没解决
└── 需要帮忙
```

### 第三步：更新进度文档

```bash
# 更新 PROGRESS.md
# 记录今天的进度
```

## 四、每周同步

### 每周例会 (30分钟，建议周末)

```
时间: 每周日下午 或 晚上
参与: 两人都要参加

议程:
1. 回顾本周完成的内容 (10分钟)
   ├── A 分享做了什么
   └── B 分享做了什么

2. 讨论遇到的问题 (10分钟)
   ├── 什么问题没解决
   ├── 需要怎么处理
   └── 谁来负责解决

3. 计划下周任务 (10分钟)
   ├── 下周要完成什么
   ├── 里程碑是什么
   └── 需要怎么配合
```

### 每周任务分配

```markdown

| 项目模板 | B | ⚪ 未开始 |
```

## 五、异常处理

### 对方不在线怎么办

```
情况1: 需要修改接口
├── 先在群里留言说明
├── 可以先写个草稿
└── 等对方上线后讨论

情况2: 发现对方的Bug
├── 在群里说明问题
├── 创建一个Issue记录
├── 尝试自己修复（如果是小问题）
└── 修复后告诉对方

情况3: 需要对方测试
├── 在群里请对方帮忙
├── 说明怎么测试
└── 等待对方反馈
```

### 代码冲突太多

```
处理步骤:

1. 先不要慌，这是正常的

2. 找个时间一起解决
   ├── 约30分钟到1小时
   ├── 共享屏幕（腾讯会议）
   └── 逐个解决冲突

3. 解决方法参考 GITHUB.md

4. 预防措施:
   ├── 每天开始前先拉取
   ├── 小步提交
   └── 修改不同文件
```

### 进度落后

```
如果一个人进度落后了:

1. 互相理解，可能有其他事情

2. 重新分配任务
   ├── 进度快的人多承担一些
   └── 调整里程碑时间

3. 提供帮助
   ├── 代码Review时多指导
   └── 分享经验和方法

4. 更新 PROGRESS.md
   └── 实时记录最新进度
```

## 💡 高效协作技巧

### 技巧1: 快速沟通

```
用简洁的方式说明问题

❌ 不好:
"那个...我这里有个问题，就是那个...你知道吗...就是..."

✅ 好:
"[detector.ts:45] detectProxy() 报错了，谁能看一下？"
```

### 技巧2: 共享进度

```
在群设置共享文档

┌─────────────────────────────────────────────────────────────┐
│  [共享表格链接]                                              │
│                                                              │
│  每个人更新自己的行:                                         │
│  ├── 今天做了什么                                            │
│  ├── 明天要做什么                                            │
│  └── 遇到什么问题                                            │
└─────────────────────────────────────────────────────────────┘
```

### 技巧3: Code Review 及时处理

```
当对方提交了 Pull Request:

✅ 尽快审查 (1小时内)
❌ 不要拖延 (不要等到第二天)

审查时:
- 先看整体，再看细节
- 发现问题及时标注
- 有疑问及时询问
```

### 技巧4: 文档同步更新

```
完成一个功能后:

1. 更新 README.md (如果需要)
2. 更新 PROGRESS.md
3. 更新自己的接口文档
4. 提交并推送
```

## 🚨 紧急情况处理

### 服务器崩溃

```
1. 立即联系对方
2. 一起查看日志
3. 确定问题原因
4. 谁的问题谁修复
5. 修复后通知对方
```

### 发现安全漏洞

```
1. 立即通知对方
2. 暂停相关功能
3. 一起评估风险
4. 尽快修复
5. 通知所有用户
```

### 数据丢失

```
1. 立即停止操作
2. 联系对方
3. 尝试从备份恢复
4. 检查 Git 历史
5. 预防下次发生
```
